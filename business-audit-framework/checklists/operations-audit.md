# Operations Audit Checklist

**Relay▸Launch Business Audit Framework**

This checklist evaluates the operational backbone of a business — the systems, processes, and practices that determine whether work gets done reliably, efficiently, and at scale. Operations problems rarely announce themselves loudly. They show up as missed deadlines, repeated mistakes, employee frustration, and a persistent feeling that the business is "always behind." This audit finds the structural causes.

## How to Use This Checklist

Score each item from 1 to 5 using the scoring rubric in `../scoring/scoring-rubric.md`. For any item scored 3 or below, add a specific note describing the gap and its observable impact on the business. Do not skip items — if an item does not apply, note why and mark it N/A.

## Category 1: Workflow Efficiency

Evaluate how work moves through the business from intake to delivery.

| # | Audit Item | Score (1-5) | Notes |
|---|-----------|-------------|-------|
| 1.1 | There is a clearly defined workflow for how new work enters the business (lead intake, project request, service ticket, etc.) | | |
| 1.2 | Work items have defined stages (e.g., Received, In Progress, Review, Complete) that are visible to all relevant team members | | |
| 1.3 | Handoffs between team members or departments have explicit triggers and checklists — work does not stall waiting for someone to notice it | | |
| 1.4 | The average time from work intake to delivery is tracked and known | | |
| 1.5 | There is a system for prioritizing work when multiple tasks compete for the same resources | | |
| 1.6 | Recurring tasks (invoicing, reporting, client check-ins) are automated or scheduled rather than relying on memory | | |
| 1.7 | The business can identify its three longest-running or most-delayed process types and has attempted to address them | | |
| 1.8 | Client or customer communication follows a defined cadence rather than happening reactively | | |
| 1.9 | Internal meetings have agendas, time limits, and documented outcomes or action items | | |
| 1.10 | The ratio of productive work to administrative overhead is understood and actively managed | | |

## Category 3: Tool Utilization

Evaluate whether the business's technology stack is serving it or creating drag.

| # | Audit Item | Score (1-5) | Notes |
|---|-----------|-------------|-------|
| 3.1 | The business has a defined technology stack and can list every tool it pays for, what each one does, and who uses it | | |
| 3.2 | There is no significant overlap between tools (e.g., two project management platforms, three communication channels serving the same purpose) | | |
| 3.3 | The primary project management or task tracking tool is used consistently by the entire team, not just the owner or manager | | |
| 3.4 | CRM or client management data is current, complete, and actively used for decision-making (not a graveyard of stale records) | | |
| 3.5 | Financial tools (accounting software, invoicing, expense tracking) are reconciled monthly at minimum | | |
| 3.6 | Communication tools have defined purposes (e.g., Slack for internal quick questions, email for client communication, project tool for task-specific discussion) | | |
| 3.7 | Integrations or automations connect key tools to reduce manual data entry (e.g., form submission creates CRM record, invoice triggers accounting entry) | | |
| 3.8 | The team has received adequate training on the tools they are expected to use | | |
| 3.9 | Tool costs are reviewed at least annually and unused subscriptions are cancelled | | |
| 3.10 | Data is backed up regularly and the business has verified that backups can be restored | | |

## Category 4: Bottleneck Identification

Evaluate whether the business understands and addresses its constraints.

| # | Audit Item | Score (1-5) | Notes |
|---|-----------|-------------|-------|
| 4.1 | The business can identify its current primary bottleneck — the single constraint that most limits throughput | | |
| 4.2 | The owner or leadership is not the approval bottleneck for routine decisions (purchases under a threshold, scheduling, standard client responses) | | |
| 4.3 | Client work does not regularly stall waiting for internal resources, information, or decisions | | |
| 4.4 | There is no single team member whose absence would halt a critical business function with no backup plan | | |
| 4.5 | Vendor or supplier dependencies are identified and alternatives exist for critical inputs | | |
| 4.6 | Feedback loops exist so that recurring problems are surfaced and addressed rather than worked around | | |
| 4.7 | The business tracks where time is actually spent (not just where it is supposed to be spent) and uses this data to identify inefficiencies | | |
| 4.8 | Scope creep on client work is identified and addressed through a change management process rather than absorbed silently | | |
| 4.9 | The business distinguishes between urgent and important work, and important work is not perpetually deferred | | |
| 4.10 | Post-project or post-incident reviews are conducted to capture lessons learned and prevent recurrence | | |

## Category 5: Capacity Planning

Evaluate whether the business understands its current capacity and plans for future demand.

| # | Audit Item | Score (1-5) | Notes |
|---|-----------|-------------|-------|
| 5.1 | The business knows its current maximum capacity — the most work it can handle without quality degradation | | |
| 5.2 | Current utilization rate is tracked (actual billable or productive hours versus available hours) | | |
| 5.3 | There is a trigger point defined for when to hire, contract, or expand resources (not reactive "we're drowning" hiring) | | |
| 5.4 | Revenue-per-team-member or revenue-per-hour metrics are known and tracked over time | | |
| 5.5 | The business has a plan for handling demand spikes (seasonal surges, viral growth, large client onboarding) | | |
| 5.6 | Workload distribution across team members is visible and roughly balanced (no one person carrying 60% of delivery while others are at 30%) | | |
| 5.7 | The business has identified which tasks should be delegated, automated, or eliminated as it scales | | |
| 5.8 | There is a forecasting process — even informal — for predicting workload 30, 60, and 90 days out | | |
| 5.9 | The cost of delivering each service or product is understood, including the labor component | | |
| 5.10 | Growth would not require a fundamental rebuild of current systems — existing infrastructure can handle 2x current volume with incremental adjustments | | |

## Interpreting Results

- **4.0 – 5.0**: Operations are a competitive advantage. Focus on optimization and scaling.
- **3.0 – 3.9**: Operations are functional but fragile. Targeted improvements will unlock capacity and reduce risk.
- **2.0 – 2.9**: Operations are a drag on the business. Structural changes are needed before growth initiatives will stick.
- **1.0 – 1.9**: Operations are in crisis. Stabilization is the priority before any other strategic work.

## Common Findings and Quick Wins

1. **No SOPs exist** — Start with the three most repeated processes. Document them this week.
2. **Owner is the bottleneck** — Define decision thresholds. Anything under $500 or 2 hours of work gets decided without owner approval.
3. **Tools are underused** — Pick one tool and commit to using it fully for 30 days before buying another one.
4. **No capacity visibility** — Implement a simple time-tracking habit for two weeks to establish a baseline.
5. **Meetings without outcomes** — Require every meeting to end with documented action items, owners, and deadlines.
